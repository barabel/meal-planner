# ADR-001: Слоистая архитектура бэкенда

## Статус
Принято

## Контекст
Electron main process нуждается в хранении и обработке данных. Можно было написать всё прямо в IPC-обработчиках, но хотелось изучить бэкенд-архитектуру (DDD, onion).

## Решение
Разделить бэкенд на 4 слоя:
- `domain/` — интерфейсы и типы (`IRecipeRepository`)
- `services/` — бизнес-логика, зависит только от доменных интерфейсов
- `repository/` — конкретная реализация хранения данных
- `controllers/` — IPC-обработчики, регистрируют `ipcMain.handle`

Сервисы получают репозиторий через constructor injection:
```ts
export class RecipeService {
  constructor(private readonly repository: IRecipeRepository) {}
}
```

## Альтернативы
- Всё в контроллерах (проще, но не тестируемо и не масштабируемо)
- Event-driven архитектура внутри main process

## Последствия
**Плюсы:** сервисы тестируемы без реального хранилища; легко сменить реализацию репозитория (с JSON на SQLite или API)

**Минусы:** больше файлов и boilerplate для простой CRUD-операции

**Когда пересмотреть:** если проект остаётся простым и слои создают лишнюю сложность без реальной пользы
